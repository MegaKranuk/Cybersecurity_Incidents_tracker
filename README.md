# Трекер інцидентів кібербезпеки

Вебзастосунок для реєстрації, моніторингу та управління кіберінцидентами.
Робота виконана в межах лабораторної роботи №3 (Варіант 2).

## Реалізований функціонал
- **Створення:** введення даних про інцидент з валідацією;
- **Редагування:** можливість зміни вже створених записів;
- **Видалення:** видалення неактуальних інцидентів;
- **Сортування та фільтрація:** сортування за датою, фільтр за типом та критичністю;
- **Збереження:** дані зберігаються у вбудованій реляційній базі даних **SQLite** з використанням власних міграцій;
- **Відображення:** динамічний рендер таблиці.

## Схема бази даних (SQLite)
База даних ініціалізується автоматично. Схема складається з 3 таблиць:
1. **`Reporters`** — `id` (TEXT PK), `name` (TEXT UNIQUE).
2. **`Incidents`** — `id` (TEXT PK), `reporterId` (TEXT FK -> `Reporters`, ON DELETE RESTRICT), `date`, `tag`, `criticality` (обмеження CHECK).
3. **`Comments`** — `id` (TEXT PK), `incidentId` (TEXT FK -> `Incidents`, ON DELETE CASCADE), `text`.
*Зв'язки реалізовано через FOREIGN KEY. Вибірка списку працює через JOIN. Для оптимізації сортування додано індекс на поле `date`.*

## Валідація введених даних
- дата інциденту — обовʼязкова, не пізніше поточної;
- тип інциденту — обовʼязкове поле;
- критичність — обовʼязкове поле;
- імʼя репортера — від 5 до 15 символів;
- опис інциденту — від 15 до 50 символів.

### Захист від XSS (Cross-Site Scripting)
Оскільки система призначена для реєстрації кіберінцидентів, користувачі можуть вводити у поля "Опис" або "Репортер" приклади шкідливого коду (payloads).
Щоб запобігти виконанню цього коду в браузері (XSS-атака), реалізовано механізм **Input Sanitization**.
* Перед рендерингом таблиці всі дані проходять через функцію-санітайзер.
* Спеціальні символи замінюються на безпечні HTML-сутності (`<` на `&lt;`, `>` на `&gt;`, `&` на `&amp;`).

## Використані технології
- **HTML5:** семантична розмітка;
- **CSS3:** Flexbox, змінні, адаптивність;
- **JavaScript:** DOM API, Event Delegation, Fetch API;
- **Backend:** Node.js, Express, TypeScript, **SQLite** (сирі SQL-запити, UUID).

## Запуск проєкту
1. Склонуйте репозиторій.
2. **Запуск сервера:** перейдіть у папку бекенду, виконайте `npm install`, а потім `npm run dev` (API запуститься на порту 3000, автоматично створиться файл `app.db` та виконаються SQL-міграції).
3. **Запуск клієнта:** відкрийте файл `index.html` у браузері (або використовуйте розширення Live Server у VS Code).

##  Сценарії тестування (curl-команди)

Для перевірки якості роботи API згідно з методичними вказівками:

1. **Отримання списку (GET із JOIN-запитом):**
   `curl -X GET "http://localhost:3000/api/incidents"`

2. **Отримання аналітики (GET із агрегацією COUNT / GROUP BY):**
   `curl -X GET "http://localhost:3000/api/incidents/stats"`

3. **Створення нового інциденту (POST - транзакційний запис у 3 таблиці):**
   `curl -X POST "http://localhost:3000/api/incidents" -H "Content-Type: application/json" -d "{\"date\":\"2026-03-28\",\"tag\":\"Фішинг\",\"criticality\":\"Середня критичність\",\"reporter\":\"Andriy Nyzhnyk\",\"comment\":\"Тестовий опис інциденту довжиною понад 15 символів\"}"`

4. **Перевірка валідації/БД (очікувана помилка 400):**
   `curl -i -X POST "http://localhost:3000/api/incidents" -H "Content-Type: application/json" -d "{\"date\":\"2026-03-28\",\"tag\":\"Фішинг\",\"criticality\":\"Неможлива критичність\",\"reporter\":\"Tester\",\"comment\":\"Тестовий опис інциденту\"}"`

5. **Демонстраційний SQLi-ендпойнт (Запит з WHERE + ORDER BY + LIMIT):**
   `curl -X GET "http://localhost:3000/api/incidents/search-vulnerable?q=' OR 1=1 --"`
