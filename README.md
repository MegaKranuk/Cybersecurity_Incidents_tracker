Cybersecurity Incident Tracker
Цей проєкт розроблено в межах лабораторної роботи №4 з дисципліни "Основи архітектури програмного забезпечення".
Застосунок дозволяє реєструвати, переглядати та редагувати інформацію про інциденти кібербезпеки з використанням клієнт-серверної архітектури.

Архітектура проєкту
Проєкт складається з двох незалежних частин:

Фронтенд (Frontend): Написаний на чистому TypeScript з використанням збирача Vite. Архітектура розділена на шари: мережевий (apiClient.ts), відображення (ui.ts), кешування (cache.ts) та керування станом (main.ts).

Бекенд (Backend): Написаний на Node.js (Express) + TypeScript. Використовує базу даних SQLite з автоматичними міграціями. Архітектура побудована за принципом розслоєння: Routes -> Controllers -> Services -> Repositories.

Інструкція із запуску (Локально)
Для запуску проєкту необхідні встановлені Node.js (v18+) та npm.

1. Запуск Бекенду
Бекенд запускається на порту 3000 і використовує локальну базу SQLite, яка створюється автоматично.

Bash
# Перейдіть у папку бекенду
cd backend

# Встановіть залежності
npm install

# Запустіть сервер у режимі розробника
npm run start:dev
Після запуску документація Swagger буде доступна за адресою: http://localhost:3000/api-docs

2. Запуск Фронтенду
Фронтенд запускається через Vite на порту 5173.

Bash
# Перейдіть у папку фронтенду
cd frontend

# Встановіть залежності
npm install

# Запустіть сервер для розробки
npm run dev
Відкрийте у браузері адресу, яку видасть Vite (зазвичай http://localhost:5173).

🛠️ Сценарії тестування API (через cURL)
Нижче наведено приклади запитів для перевірки працездатності серверної частини згідно з вимогами лабораторної роботи.

1. Отримання списку інцидентів (GET)
Перевірка базового отримання даних з підтримкою пагінації:

Bash
curl -X GET "http://localhost:3000/api/v1/incidents?page=1&pageSize=5" -H "accept: application/json"
2. Створення нового інциденту (POST)
Перевірка створення ресурсу. Очікується статус 201 Created:

Bash
curl -X POST "http://localhost:3000/api/v1/incidents" \
-H "Content-Type: application/json" \
-d '{
  "date": "2026-04-25",
  "tag": "DDoS-атака",
  "criticality": "Дуже критично",
  "reporter": "Адміністратор",
  "comment": "Виявлено аномальний трафік на порт 443"
}'
3. Перевірка обробки помилок валідації (400 Bad Request)
Спроба відправити занадто коротке ім'я репортера або відсутню дату. Очікується деталізована помилка у форматі {"error": {...}}:

Bash
curl -X POST "http://localhost:3000/api/v1/incidents" \
-H "Content-Type: application/json" \
-d '{
  "tag": "Фішинг",
  "criticality": "Низька критичність",
  "reporter": "A", 
  "comment": "Занадто короткий коментар"
}'
4. Перевірка CORS (OPTIONS)
Оскільки фронтенд знаходиться на іншому порту, можна перевірити, як сервер обробляє preflight-запити браузера:

Bash
curl -X OPTIONS "http://localhost:3000/api/v1/incidents" \
-H "Origin: http://localhost:5173" \
-H "Access-Control-Request-Method: POST" \
-i
Очікується статус 204 No Content та наявність заголовка Access-Control-Allow-Origin.

🤝 Контракти даних (DTO)
Проєкт використовує сувору типізацію DTO. Для підтримки зворотної сумісності (версія API v1):

Заборонено перейменовувати або видаляти базові поля (id, date, tag, criticality).

Нові поля (наприклад, reporterId) додаються виключно як опціональні.
